# SPEC-API-001 구현 계획

## 개요

본 문서는 SPEC-API-001 요구사항(REQ-API-001 ~ REQ-API-019)의 구현 접근법, 마일스톤,
기술 스택 결정, 리스크를 정의한다. 구현 코드는 `/moai:2-run` 단계에서 작성되며,
본 문서는 계획 문서이다.

## 3. 마일스톤 (우선순위 기반)

### Milestone 1: 환경 변수 인프라 (Priority High)

**목표**: dev/staging/prod 환경 분리 및 빌드 시점 변수 주입 확립.

- REQ-API-016: `.env.example`, `.env`, `.env.staging`, `.env.production` 파일 구조 생성
- REQ-API-017: `app.config.ts`에서 환경 변수 읽어 `extra` 필드에 주입
- REQ-API-018: `src/config/env.ts`에서 런타임 검증 (fail-fast)
- REQ-API-019: EAS Build 프로필별 환경 변수 분리 검증

**완료 조건**: 각 환경(dev/staging/prod)에서 올바른 Supabase URL이 주입되고,
누락 시 초기화 에러가 발생함을 테스트로 검증.

### Milestone 2: Supabase 클라이언트 싱글톤 (Priority High)

**목표**: 앱 전역 재사용 가능한 타입 안전 클라이언트 인스턴스 확립.

- `@supabase/supabase-js`, `expo-secure-store`, `expo-constants` 의존성 추가
- REQ-API-001: `src/lib/supabase.ts` 싱글톤 인스턴스 생성
- REQ-API-002: 세션 영속화(SecureStore 우선, AsyncStorage 폴백), 자동 갱신 설정
- REQ-API-003: Realtime 채널 API 접근성 검증
- REQ-API-004: `invokeEdgeFunction` 래퍼 구현
- REQ-API-005: 환경 변수 누락 시 fail-fast 검증

**완료 조건**: 클라이언트가 초기화되고, 세션이 영속화되며, 모든 기능(from, channel,
functions, storage, auth)에 접근 가능함을 테스트로 검증.

### Milestone 3: gen-types 타입 안전성 (Priority High)

**목표**: 12개 엔터티 + ENUM + 보안 뷰의 TypeScript 타입 자동 생성.

- REQ-API-006: `supabase gen types typescript` 실행 → `src/types/db.ts` 생성
- REQ-API-007: `createClient<Database>()` 제네릭 적용
- REQ-API-008: 12개 엔터티별 도메인 타입 별칭 정의 (`src/types/index.ts`)
- REQ-API-009: `sticker_type` ENUM 타입 노출 검증
- REQ-API-010: `user_profiles`, `user_books_public` 뷰 타입 노출 검증

**완료 조건**: `supabase.from('users').select()` 반환 타입이 `User[]`로 추론되고,
존재하지 않는 테이블명 입력 시 컴파일 에러가 발생함을 검증.

### Milestone 4: 공통 에러 처리 (Priority Medium)

**목표**: 일관된 에러 분류, 재시도, 사용자 친화적 메시지 제공.

- REQ-API-011: `normalizeError` 함수 구현
- REQ-API-012: 7개 에러 카테고리 분류 로직
- REQ-API-013: 지수 백오프 재시도 래퍼 (NETWORK/SERVER만, 최대 3회)
- REQ-API-014: 한국어 사용자 친화적 메시지 매핑 테이블
- REQ-API-015: 구조화 로깅 (Sentry `captureException` 호환 포맷)

**완료 조건**: 네트워크 에러가 3회 재시도 후 적절한 카테고리로 분류되고,
한국어 메시지가 반환됨을 테스트로 검증.

## 4. 기술 접근 (Technical Approach)

### 4.1 클라이언트 싱글톤 패턴

ES 모듈 캐싱을 활용한 싱글톤. `src/lib/supabase.ts` 최상위에서 `createClient()` 호출
후 `export`. React Native Metro 번들러의 모듈 캐시가 동일 참조를 보장한다.

세션 저장소는 `expo-secure-store` 어댑터를 우선 사용하고, SecureStore 용량 제한(2KB)에
걸리면 `@react-native-async-storage/async-storage`로 폴백하는 커스텀 어댑터 구현
(Open Question 6.3 해결 방안).

### 4.2 gen-types 워크플로

```bash
# 1. Supabase CLI로 타입 생성
supabase gen types typescript --project-id <project-ref> > src/types/db.ts

# 2. 스키마 변경 시마다 재실행 (SPEC-DB-001 migration 후)
```

`src/types/db.ts`는 `.gitignore`에서 제외(커밋 대상) — 타입 안정성 보장 및 CI 환경에서
Supabase CLI 의존성 감소. 단, 파일 상단에 `// AUTO-GENERATED BY supabase gen types.
DO NOT EDIT MANUALLY.` 주석 필수.

### 4.3 에러 처리 아키텍처

```
Supabase 에러 (PostgREST / Auth / Network)
        │
        ▼
  normalizeError()          ← 에러 객체 → ApiError 변환
        │
        ▼
  classifyError()           ← 7개 카테고리 분류
        │
        ├─ NETWORK / SERVER → retryWithBackoff() (최대 3회, 1s/2s/4s)
        │
        ├─ AUTH → 즉시 반환 (SPEC-AUTH-001이 세션 갱신 처리)
        │
        ├─ RLS_DENIED → 즉시 반환 + 사용자 메시지
        │
        └─ VALIDATION / NOT_FOUND / UNKNOWN → 즉시 반환
        │
        ▼
  getUserFriendlyMessage()  ← 한국어 사용자 메시지 매핑
        │
        ▼
  logToSentry()             ← UNKNOWN 및 재시도 실패만 로깅
```

### 4.4 환경 변수 주입 체인

```
.env / .env.staging / .env.production
        │ (EAS Build profile에 따라 선택)
        ▼
  app.config.ts             ← process.env.EXPO_PUBLIC_* 읽기
        │
        ▼
  Expo Config (extra 필드)  ← { supabaseUrl, supabaseAnonKey, ... }
        │
        ▼
  Constants.expoConfig.extra ← 런타임 접근
        │
        ▼
  src/config/env.ts         ← 검증 (fail-fast) + 접근 함수
        │
        ▼
  src/lib/supabase.ts       ← createClient(url, key)
```

## 6. 리스크 및 대응

### 리스크 1: SecureStore 용량 제한 (낮음)

**문제**: iOS Keychain 단일 항목 2KB 제한. Supabase JWT + 리프레시 토큰 조합이 제한 초과 가능.

**대응**: 커스텀 저장소 어댑터에서 SecureStore 실패 시 AsyncStorage 폴백. Open Question
6.3 해결 방안으로 문서화. `/moai:2-run` 단계에서 실제 토큰 길이 측정 후 결정.

### 리스크 2: gen-types와 스키마 불일치 (중간)

**문제**: SPEC-DB-001 migration 배포 후 gen-types 재실행 누락 시 타입 불일치 발생.

**대응**: CI 파이프라인(SPEC-DEPLOY-001)에 gen-types 재실행 검증 스텝 추가 권장.
`src/types/db.ts` 파일 상단에 생성 시간 및 스키마 해시 주석 포함하여 추적성 확보.

### 리스크 3: 데이터 페칭 전략 미확정 (중간)

**문제**: Open Question 6.1(React Query vs SWR vs 순수 훅) 미해결 상태로 도메인 SPEC
구현이 지연될 수 있음.

**대응**: 본 SPEC은 클라이언트 인프라만 제공하므로 즉시 구현 가능. 데이터 페칭 결정은
SPEC-AUTH-001 또는 SPEC-BOOK-001 구현 시작 시점에 확정. 각 도메인 SPEC은 `supabase`
클라이언트를 직접 사용하므로 데이터 페칭 라이브러리 의존성 없이도 동작.

### 리스크 4: 환경 변수 노출 (낮음)

**문제**: `EXPO_PUBLIC_*` 변수는 클라이언트 번들에 평문 포함. `SUPABASE_ANON_KEY`가
노출되나, RLS가 데이터를 보호하므로 설계상 허용됨 (SPEC-DB-001 가정 2.2.2).

**대응**: `service_role` 키는 절대 `EXPO_PUBLIC_` 접두사 없음. `.env` 파일 `.gitignore`
확정. 코드 리뷰에서 `service_role` 키 사용 시 빌드 차단.

### 리스크 5: Realtime 연결 안정성 (낮음)

**문제**: SPEC-FEED-001 피드 실시간 업띡데이트에 사용되는 Realtime 채널이 네트워크
불안정 시 끊김.

**대응**: `@supabase/supabase-js` v2는 자동 재연결 로직 내장. 본 SPEC은 채널 API
접근성만 보장하고, 재연결 정책은 라이브러리 기본값 사용. SPEC-FEED-001 구현 시
재연결 핸들링 커스터마이즈 가능.

## 7. 테스트 전략

### 7.1 단위 테스트 (Jest)

- **클라이언트 싱글톤**: 동일 참조 반환 검증, 초기화 에러(fail-fast) 검증
- **에러 정규화**: 7개 카테고리 분류 정확도, 한국어 메시지 매핑
- **재시도 로직**: NETWORK/SERVER 카테고리만 재시도, 지수 백오프 간격, 최대 3회
- **환경 변수 검증**: 누락 시 에러 발생, 정상 시 값 반환

### 7.2 통합 테스트 (Supabase 로컬 개발 서버)

- **gen-types 타입 정확성**: 12개 엔터티 Row 타입이 실제 컬럼과 일치
- **세션 영속화**: 앱 재시작 후 세션 복원
- **Edge Function 호출**: `invokeEdgeFunction` 래퍼 동작

### 7.3 타입 테스트 (컴파일 시점)

- 존재하지 않는 테이블명 → 컴파일 에러
- 잘못된 컬럼명 → 컴파일 에러
- ENUM 값 외 입력 → 컴파일 에러

## 8. 선행/후속 SPEC 연동

### 선행 (이미 완료)
- **SPEC-DB-001** (v1.2.0): 12개 엔터티 스키마, RLS, 트리거, 보안 뷰 — 본 SPEC의 타입 기반
- **SPEC-UI-001** (v1.0.0): 프로젝트 구조, TypeScript strict, Jest 설정 — 본 SPEC의 빌드 환경

### 후속 (본 SPEC 완료 후 시작)
- **SPEC-AUTH-001**: OAuth 로그인 — `supabase.auth.signInWithOAuth()` 사용
- **SPEC-NAV-001**: 네비게이션 — 인증 상태 기반 라우팅
- **SPEC-BOOK-001** ~ **SPEC-PROFILE-001**: 모든 도메인이 `supabase` 클라이언트 사용
- **SPEC-DEPLOY-001**: EAS Build 환경 분리, Sentry 통합 — 본 SPEC의 환경 변수 체인 활용
